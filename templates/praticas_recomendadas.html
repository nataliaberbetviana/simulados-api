Práticas de Segurança AWS

Melhores Práticas para Criar e Proteger sua Conta AWS

📋 Por Que Segurança é Fundamental?

A segurança da sua conta AWS é absolutamente crítica. Uma conta comprometida pode resultar em custos inesperados massivos, perda de dados, exposição de informações confidenciais e danos à reputação da empresa. A AWS opera sob o modelo de responsabilidade compartilhada: AWS é responsável pela segurança DA nuvem, você é responsável pela segurança NA nuvem.

Implementar práticas de segurança desde o primeiro dia não é opcional - é essencial. Este guia cobre as melhores práticas recomendadas pela AWS para criar e gerenciar sua conta com segurança máxima, desde a configuração inicial até operações do dia a dia.

🎯 Práticas Críticas Imediatas

1. Proteja a Conta Root IMEDIATAMENTE

O QUE FAZER: Assim que criar sua conta AWS, configure MFA (Multi-Factor Authentication) na conta root. Use autenticador de hardware ou aplicativo (Google Authenticator, Authy).

POR QUÊ: A conta root tem acesso IRRESTRITO a tudo. Se comprometida, atacante pode excluir recursos, criar instâncias caras, acessar dados sensíveis.

NUNCA: Use a conta root para tarefas diárias. Guarde as credenciais em local extremamente seguro (cofre físico, gerenciador de senhas corporativo).

2. Crie Usuário IAM Administrativo

O QUE FAZER: Imediatamente após proteger conta root, crie usuário IAM com permissões administrativas para uso diário.

COMO: Acesse IAM Console → Users → Add User → Anexe política AdministratorAccess → Habilite MFA para este usuário também.

IMPORTANTE: Use este usuário administrativo para tarefas diárias, NUNCA a conta root. Conte root deve ser usada apenas para tarefas que EXIGEM root (fechar conta, alterar plano de suporte, etc).

3. Configure Informações de Contato Alternativo

O QUE FAZER: Configure email alternativo, telefone e contatos de segurança/operações na conta.

POR QUÊ: Se perder acesso ao email principal, AWS pode usar contatos alternativos para recuperação. Em emergências de segurança, AWS precisa forma de contatar você urgentemente.

ONDE: Account Settings → Alternate Contacts → Configure Security, Billing e Operations contacts.

4. Ative CloudTrail Imediatamente

O QUE FAZER: Crie trilha CloudTrail em todas as regiões para registrar TODAS as ações na conta.

POR QUÊ: Sem CloudTrail, você não tem registro de quem fez o quê e quando. Essencial para auditoria, compliance, investigação de incidentes e forense.

CONFIGURE: Multi-região trail, valide integridade de logs, criptografe com KMS, armazene em bucket S3 dedicado com versionamento e bloqueio.

🔑 Gerenciamento de Identidade e Acesso

Princípio do Privilégio Mínimo

Conceda apenas as permissões estritamente necessárias para realizar tarefa.
Comece sem permissões e adicione conforme necessário.
Nunca comece com AdministratorAccess e "remova depois".
Revise e reduza permissões regularmente.
Use AWS Access Analyzer para identificar permissões não utilizadas.

Use Grupos IAM, Não Usuários Individuais

Crie grupos para diferentes funções (Desenvolvedores, DevOps, Auditores, Somente Leitura).
Atribua permissões aos grupos, não usuários.
Adicione usuários aos grupos apropriados.
Facilita gerenciamento e garante consistência.
Quando pessoa muda de função, apenas mude grupo.

Implemente Política de Senha Forte

Configure política de senha IAM: mínimo 14 caracteres, exija letras maiúsculas, minúsculas, números e símbolos.
Exija alteração a cada 90 dias.
Impeça reutilização das últimas 24 senhas.
Permita que usuários alterem próprias senhas.
IAM Console → Account Settings → Password Policy.

MFA para TODOS os Usuários Privilegiados

Exija MFA para conta root (crítico), usuários administrativos (essencial), todos os usuários IAM (altamente recomendado).
Use autenticadores de hardware (YubiKey) para contas mais sensíveis.
Aplicativos autenticadores (Google Authenticator, Authy) para demais.
NUNCA use SMS como único fator.

Rotação Regular de Credenciais

Senhas: Política de expiração 90 dias automaticamente.
Chaves de Acesso: Rotacione a cada 90 dias manualmente ou automaticamente.
Use Credential Report para identificar chaves antigas. Desative chaves não utilizadas.
NUNCA commite chaves em código ou repositórios Git.

Use IAM Roles em Vez de Chaves para Aplicações

Para aplicações em EC2, Lambda, ECS: use IAM Roles, NUNCA chaves hardcoded.
Roles fornecem credenciais temporárias rotacionadas automaticamente.
Elimina risco de expor chaves em código.
Para aplicações externas: use AWS STS para credenciais temporárias quando possível.

🛡️ Configurações de Rede e Infraestrutura

Nunca Exponha Recursos Diretamente à Internet

VPC (Virtual Private Cloud) para isolar recursos.
Coloque recursos sensíveis em subnets privadas sem acesso direto à internet.
Use NAT Gateway para acesso outbound de subnets privadas.
Apenas load balancers e bastion hosts em subnets públicas. Banco de dados, servidores de aplicação SEMPRE em subnets privadas.

Grupos de Segurança: Princípio do Menor Acesso

NUNCA permita 0.0.0.0/0 em portas sensíveis (SSH 22, RDP 3389, bancos de dados).
Permita apenas IPs específicos necessários.
Use referências de Security Group (sg-xxxxx) em vez de ranges de IP quando possível.
Documente motivo de cada regra usando tags e descriptions.
Revise regras regularmente e remova as não utilizadas.

Network ACLs como Camada Adicional

Use Network ACLs como segunda camada de defesa.
Security Grouuls: stateful, nível de instância. NACLs: stateless, nível de subnet.
Configure NACLs para bloquear tráfego malicioso conhecido.
Útil para regras de negação explícitas (Security Groups são apenas allow)

.

Ative VPC Flow Logs

Registre todo tráfego de rede aceito e rejeitado em sua VPC.
Essencial para análise de segurança, troubleshooting, detecção de anomalias.
Envie logs para CloudWatch Logs ou S3.
Use ferramentas como Amazon Detective ou GuardDuty para análise automatizada de padrões suspeitos

Use AWS WAF para Aplicações Web

Proteja aplicações web contra ataques comuns (SQL injection, XSS, bots).
Configure WAF em ALB, CloudFront ou API Gateway
Use AWS Managed Rules para proteção básica.
Crie regras personalizadas para proteção específica da aplicação.
Monitore métricas e logs WAF continuamente.

📊 Monitoramento e Detecção

Ative Amazon GuardDuty

Serviço de detecção de ameaças que analisa CloudTrail, VPC Flow Logs e DNS Logs.
Usa machine learning para identificar atividades maliciosas.
Detecta comprometimento de credenciais, comunicação com IPs maliciosos, reconhecimento de infraestrutura, mineração de criptomoedas. Ative em TODAS as regiões e contas.

Configure AWS Config para Compliance Contínuo

Registre configurações de recursos e mudanças ao longo do tempo.
Implemente Config Rules para conformidade automática (ex: S3 buckets não podem ser públicos, EC2 deve ter encryption habilitada).
Configure remediação automática para violações não críticas.
Gere relatórios de compliance para auditoria.

Use AWS Security Hub

Agregação centralizada de descobertas de segurança de GuardDuty, Inspector, Macie, IAM Access Analyzer, Firewall Manager.
Avalia conta contra frameworks de segurança (CIS AWS Foundations, PCI-DSS).
Prioriza descobertas por severidade.
Integra com SIEM externo se necessário.

Configure Alertas CloudWatch

Crie alarmes para atividades críticas: uso de conta root, falhas de login IAM, mudanças em Security Groups, criação de usuários IAM, mudanças em políticas IAM, criação de Access Keys, desativação de CloudTrail, mudanças em Network ACLs.
Envie alertas para SNS → email/SMS/Slack para resposta imediata.

Implemente Análise Regular de Logs

Use CloudWatch Logs Insights ou Amazon Athena para queries em logs CloudTrail e VPC Flow Logs.
Procure por padrões suspeitos: logins de IPs desconhecidos, horários incomuns, múltiplas falhas seguidas de sucesso, grandes volumes de criação/exclusão de recursos, acesso a serviços raramente usados.

💾 Proteção de Dados

Criptografia em Repouso - SEMPRE

S3: Habilite Server-Side Encryption (SSE-S3 ou SSE-KMS) por padrão em todos os buckets. Use Bucket Policies para rejeitar uploads não criptografados.

EBS: Habilite criptografia por padrão na região. Todos os novos volumes serão criptografados automaticamente.

RDS: Sempre crie databases com encryption at rest habilitada. Não pode ser habilitada depois da criação.

DynamoDB: Use criptografia com AWS KMS para dados sensíveis.

Criptografia em Trânsito - SEMPRE

Use TLS/SSL para toda comunicação: ALB/NLB com certificados SSL, CloudFront com HTTPS obrigatório, API Gateway apenas HTTPS, RDS com SSL/TLS forçado, ElastiCache com in-transit encryption.
Desabilite protocolos antigos (SSLv3, TLS 1.0, TLS 1.1).
Use apenas TLS 1.2 ou 1.3.

Gerenciamento de Chaves com AWS KMS

Use Customer Managed Keys (CMK) para controle completo sobre criptografia.
Implemente políticas de chave restritivas.
Habilite automatic key rotation anual.
Use aliases para facilitar gerenciamento. Monitore uso de chaves com CloudTrail.
NUNCA compartilhe chaves entre ambientes (dev, staging, prod).

Controle de Acesso a Dados Sensíveis

Use S3 Block Public Access em nível de conta e bucket.
Configure S3 Bucket Policies restritivas.
Implemente Object Lock para dados que não podem ser alterados (compliance, legal).
Use versionamento para recuperação de exclusões acidentais.
Configure lifecycle policies para transição e expiração automática.

Backups Regulares e Automáticos

AWS Backup: Configure planos de backup centralizados para EC2, EBS, RDS, DynamoDB, EFS, FSx, Storage Gateway.

Frequência: Diário para dados críticos, semanal para demais.

Retenção: Mínimo 30 dias, idealmente 90 dias ou mais.

Teste: Faça restore testing mensalmente para garantir que backups funcionam.

🏢 Governança e Compliance

Use AWS Organizations

Gerencie múltiplas contas AWS centralizadamente.
Crie estrutura organizacional (OUs) por ambiente, departamento ou aplicação.
Implemente Service Control Policies (SCPs) para guardrails em nível de organização.
Consolide billing.
Compartilhe recursos (AMIs, snapshots) entre contas de forma controlada.

Implemente Arquitetura Multi-Conta

Separe ambientes por conta: Produção, Staging, Desenvolvimento, Segurança, Log Archive, Shared Services.
Isolamento de blast radius: problema em dev não afeta produção.
Separação de billing e limites de serviço.
Facilita compliance e auditoria.
Use AWS Control Tower para automação de setup.

Tags Obrigatórias para Recursos

Defina política de tagging: Owner (quem é responsável), Environment (dev/staging/prod), Project (a qual projeto pertence), CostCenter (centro de custo), Compliance (nível de sensibilidade).
Use Tag Policies no Organizations para forçar conformidade.
Facilita rastreamento de custos, auditoria e gerenciamento.

Service Control Policies (SCPs)

Defina permissões máximas para contas na organização.
Exemplos: impeça desativação do CloudTrail, bloqueie criação de recursos fora de regiões aprovadas, proíba exposição pública de S3 buckets, impeça exclusão de recursos de segurança.
SCPs são guardrails, não substituem IAM policies.

Documentação e Procedimentos

Documente arquitetura de segurança, responsabilidades, procedimentos de resposta a incidentes, processos de onboarding/offboarding. Mantenha runbooks para cenários comuns.
Documente decisões de arquitetura e motivos.
Facilita auditorias, treinamento e continuidade operacional.

🚨 Resposta a Incidentes

Preparação: Plano de Resposta

Crie plano documentado de resposta a incidentes ANTES que ocorra problema.
Defina roles e responsabilidades (quem faz o quê).
Liste contatos de emergência (AWS Support, gerenciamento, legal, PR).
Documente procedimentos passo a passo para cenários comuns: credenciais comprometidas, recursos criados maliciosamente, exfiltração de dados.

Detecção: Monitore Anomalias

Use GuardDuty, Security Hub, CloudWatch para detecção.
Configure alertas para atividades suspeitas.
Estabeleça baseline de atividade normal.
Investigue prontamente qualquer desvio.
Mantenha equipe treinada para reconhecer indicadores de comprometimento.

Contenção: Ações Imediatas

Credenciais comprometidas: Desative usuário IAM imediatamente, rotacione todas as chaves, revogue sessões ativas, analise logs CloudTrail para ações realizadas.

Recurso comprometido: Isole instância (mude Security Group para negar tudo), capture snapshot/AMI para forense, NÃO delete ainda, analise logs e memória.

Conta comprometida: Abra caso com AWS Support URGENTE, mude senha root se necessário, desative todos os usuários, rotacione todas as credenciais.

Erradicação e Recuperação

Identifique e remova causa raiz do comprometimento. Remova backdoors, malware, acessos não autorizados.
Restaure sistemas a partir de backups limpos conhecidos.
Reforce configurações de segurança.
Implemente controles adicionais para prevenir reocorrência.

Lições Aprendidas: Post-Mortem

Após incidente, realize reunião de post-mortem sem culpa.
Documente: o que aconteceu, como foi detectado, quanto tempo levou para responder, o que funcionou bem, o que pode melhorar.
Crie action items específicos.
Atualize procedimentos.
Compartilhe lições com equipe.
Teste melhorias implementadas.

💰 Segurança de Billing e Custos

Configure Billing Alerts

Ative billing alerts no console de billing.
Configure alarmes CloudWatch para custos estimados.
Defina múltiplos thresholds: 50%, 75%, 90%, 100% do budget.
Receba alertas ANTES de exceder budget.
Ataques podem gerar custos massivos rapidamente (mineração de cripto, instâncias caras).

Use AWS Budgets

Crie budgets mensais por serviço, tag ou conta.
Configure alertas quando custos atingirem thresholds.
Budgets podem incluir créditos e RI coverage.
Use budget actions para automação (notificar, aplicar SCP, desligar recursos).
Revise budgets mensalmente e ajuste conforme necessário.

Implemente Service Quotas

Defina limites apropriados para serviços críticos.
Previne criação acidental ou maliciosa de recursos em massa.
Monitore uso vs quotas com Trusted Advisor.
Para cargas legítimas, solicite aumentos proativamente.
Para prevenir abuso, mantenha quotas conservadoras.

Restrinja Regiões AWS

Use SCPs para permitir recursos apenas em regiões específicas necessárias.
Reduz superfície de ataque e custos.
Facilita compliance com regulamentações de residência de dados.
Simplifica monitoramento (menos regiões para verificar).
Exceções apenas quando necessário e documentado.

⚠️ Erros Comuns a EVITAR

❌ Usar Conta Root para Operações Diárias

NUNCA use conta root depois da configuração inicial.
Crie usuário administrativo para tarefas diárias.
Conta root deve ficar guardada e usada apenas para tarefas que exigem root explicitamente.

❌ Não Implementar MFA

MFA não é opcional para contas com acesso privilegiado.
Senhas sozinhas são facilmente comprometidas.
MFA adiciona camada crítica de proteção.
Sem MFA, você está apenas um phishing de ter conta comprometida.

❌ Expor Chaves de Acesso em Código

NUNCA commite chaves AWS em Git, Dockerfile, configurações.
Use IAM Roles para aplicações em AWS.
Use secrets managers (AWS Secrets Manager, Parameter Store) para credenciais.
Bots escaneiam GitHub constantemente para chaves expostas - minutos após commit, sua conta pode estar comprometida.

❌ Grupos de Segurança Abertos (0.0.0.0/0)

NUNCA abra SSH (22), RDP (3389), bancos de dados para toda internet.
Use bastion hosts, VPN ou Systems Manager Session Manager para acesso administrativo.
Restrinja acesso a IPs específicos conhecidos quando possível.
Revise regras regularmente.

❌ Não Habilitar Logging

Sem CloudTrail, você não sabe quem fez o quê.
Sem VPC Flow Logs, não sabe que tráfego está ocorrendo.
Sem logs, investigação de incidentes é impossível.
Compliance requer logs.
Habilite logging SEMPRE e em TODAS as regiões.

❌ Ignorar Alertas de Segurança

GuardDuty, Security Hub, Trusted Advisor não são opcionais.
Alertas não são sugestões - são indicadores de problemas reais. Configure alertas para serem acionáveis.
Investigue prontamente.
Fadiga de alertas é problema real, mas solução é ajustar alertas, não ignorá-los.

❌ Não Fazer Backups

Ransomware, exclusões acidentais, falhas de hardware acontecem.
Sem backups, perda de dados é permanente.
Configure backups automáticos.
Teste restores regularmente.
Backups não testados são backups que não funcionam.

❌ Conceder Mais Permissões que Necessário

AdministratorAccess para todos é receita para desastre.
Privilégio mínimo não é sugestão, é requisito de segurança.
Comece com nenhuma permissão, adicione conforme necessário.
Use Access Analyzer para identificar permissões não utilizadas e remover.
